## What worked

The fixes shipped in d2f68c5 are doing real work for the warm-cache, same-tab-session scenario:

1. **Cache durability across 15+ min idle** (Tests 1, 2, 3, 5) — sessionStorage SWR keys survive Chrome's tab freezing/throttling. After idle, sidebar click paints the cached data instantly. No skeleton stuck.
2. **Background SWR refresh** is silent and non-blocking — the user sees cached rows immediately while the network refresh happens in the background.
3. **CRIT-1 master-fetch concurrency limit** holds — Test 4 fresh-reopen of /orders/new fired 36 fetches in clean 4-at-a-time waves, not a 25-call wall. Wave timeline 200ms / 3.8s / 4.8s / 5.0s / etc.
4. **Topbar /notifications dup fix (CRIT-3)** confirmed on each cold load — fires exactly 1× now, not 2×.

## What still breaks

### 🔴 CRIT-4 silent-fail variant (REPRODUCED on production)
**Repro:** any sidebar SPA-nav from one page to another where the destination's SWR cache has been evicted (browser memory, sessionStorage quota, manual clear, prior idle → discard cycle, etc).
**Symptom:** zero supabase requests fire, page renders empty-state ("0 suppliers" / "No customers" / etc), no skeleton, no error. The `useSWRList` fetcher promise resolves silently with an empty data set.
**Workaround:** hard reload of the same URL → fetcher fires → 50 rows in <1s.
**Root cause likely:** when `fetchAllPaged` runs after extended idle, the supabase-js client is in a subtle state where the request resolves (no exception, no XHR fired) but returns an empty result. The d2f68c5 patches (auth pre-warm, page timeout, useSWRList grace timeout) don't catch this case because the promise DOES resolve — just with empty data.

### 🔴 Chrome tab-discard chaos
Across this test, **Chrome discarded the ERP tab 5 times** during idle periods. Tab IDs went 841065097 → 841065147 → 841065150 → 841065154 → 841065157 → 841065160.
- Each discard was during a 15-min idle (test waiting cycle).
- On restore, Chrome shows the tab again with the prior URL, but the JS context is reset (fresh window object, no `window.__loop` state, fresh sessionStorage hydration from the tab record).
- This is Chrome's standard memory-pressure behavior and will hit real users on lower-end laptops.

### 🔴 ChunkLoadError after deploy across long session
**Repro:** open ERP tab. Leave open for hours. A Vercel deploy lands (could be perf-monitor cron, normal CI, etc). User clicks any sidebar link that lazy-loads a chunk (e.g. CustomersPage). The lazy import points to the OLD chunk hash, which Vercel CDN no longer serves.
**Symptom:** the App.jsx PageErrorBoundary catches it and shows "Your open tab is out of date. Click Reload."
**Workaround:** click reload.
**Notes:** the error UI is friendly, but the user has lost any in-flight unsaved form state. For an operator filling out an order over several minutes, this is data loss. There is no auto-reload or auto-recovery.

## Patterns Identified

1. **Cache survival ≠ cache always present.** Tests 1-5 succeeded because sessionStorage carried the SWR cache through idle. Test 6 failed because something — Chrome cleared it, sessionStorage hit quota, prior tab-discard reset state — left the cache empty. The d2f68c5 fix only protects the cache-survives path. The cache-evicted-then-clicked path is unprotected.

2. **Tab discarding is the most user-visible chaos.** Even when our app code is correct, Chrome's memory pressure can recycle the tab. On restore, the user often hits both bugs above (no cache → silent fail; old chunk → reload prompt).

3. **The deferred-master pre-fetch is well-behaved.** When primeMasters fires (from /orders/new), it issues fetches in proper waves of ≤4 concurrent. This is solid.

4. **Backend is innocent.** Supabase logs and Vercel logs show no errors at all during the entire 4-hour test window. Latency on returned fetches is consistently 200-700ms per page — that's healthy for paginated 1000-row queries on Supabase ap-southeast-1 from India.

## Recommendations (NOT implemented — read-only audit)

### Immediate (next sprint)

1. **Defensive guard in `useSWRList`:** if `fetcher` resolves with an empty array AND the page expects data (for masters with known non-zero row count), trigger a one-time silent retry after 1s. This catches the silent-fail mode.
2. **Service worker for chunk recovery:** intercept failed chunk requests and serve fallbacks, OR pre-cache the next-likely chunks. Apps with operator workflows should NOT lose form state to ChunkLoadError.
3. **Auto-reload on chunk fail:** in PageErrorBoundary's chunk-load case, if no unsaved form data, just reload silently after a 1s delay. Show the friendly error only when there's a form with dirty values.

### Medium

4. **Persistent-tab indicator + manual refresh prompt** — when the tab is older than ~30 min, surface a small "Refresh" pill in the Topbar. Lets users opt-in to a clean reload before encountering ChunkLoadError.
5. **Logging/telemetry on the silent-fail path** — when `useSWRList`'s fetcher returns empty for a key that historically had data, post a beacon (Sentry/equivalent). Currently this bug is invisible — we'd never know it's happening to real users.

### Structural

6. **Move list-fetching to a single source of truth** — replace per-page useSWRList + AppContext primeMasters with one unified data layer that has explicit error states. Eliminates the silent-empty failure mode entirely.

## Final Verdict (10× cycle audit)

**For warm-cache normal use: the d2f68c5 fix works.** 5 of 5 happy-path idle tests passed cleanly.

**For cache-cold sidebar nav AND tab-discard recovery scenarios: the bug remains.** The user-felt "lag/empty page" can still surface after extended idle, especially on memory-pressured systems. The d2f68c5 fix is a meaningful improvement but not a complete cure.

**Backend is healthy.** Don't waste time looking at Supabase or Vercel for the lag. The remaining failures are entirely in the React data layer's silent-empty failure mode and Chrome's tab lifecycle.

**Suggested next step:** ship recommendations 1 + 3 above. They'd cover the two reproducible failures from this cycle.
